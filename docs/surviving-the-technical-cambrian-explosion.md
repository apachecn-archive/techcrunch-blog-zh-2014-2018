# 从寒武纪科技大爆发中幸存下来

我最近做了很多招聘工作。我的雇主们正处于另一轮增长高潮中。我最有启发性的问题之一是:“你有什么策略来跟上技术框架、语言、数据库、模板系统等等的寒武纪大爆发？”每个人都有策略——但似乎没有人觉得他们的策略特别好。

一方面，对于一名软件工程师来说，这是一个令人惊奇和兴奋的时代。新的强大的工具和技术似乎每周都会出现，几乎总是*几乎*准备好迎接黄金时间。但与此同时，我们的领域已经成为一个永远令人困惑的领域。假设你正在建立一个网站。你的前端应该使用 Angular 还是 React？(而你说的有角，是指有角 1.0 还是 2.0？)那么 Ember，或者说流星呢？

至于你的后端:什么语言，什么框架？Ruby/Rails？Python/Django？一个灯堆？围棋似乎很酷。人们都说 C#很棒。Java 仍然很受欢迎，你尊敬的人对 Scala 赞不绝口，或者你甚至会对 Erlang 或 Haskell 感到难以理解。至于你的数据存储:SQL？NoSQL？两者的结合，比如说 Postgres 和 Redis？如果你有一大堆*数据怎么办？Hadoop 还值得追求吗？图形数据库怎么样，比如 Spark 上的 GraphX？*

你可能认为构建移动应用程序会容易得多。你就错了。你可以用 Objective-C 或 Swift 编写你的 iOS 应用程序……但这绝不是仅有的两种选择。Xamarin 呢，这样就可以针对*所有*移动平台了？还是反应原生？还是 PhoneGap？还是一个 Unity app？别忘了，你现在必须为半打不同尺寸的 iPhone 和 iPad 编码。至于安卓，我的意思是，连我*开始*都不要了。

我以前曾在[中半开玩笑地写过这个问题](https://web.archive.org/web/20230129062556/https://techcrunch.com/2014/10/18/you-too-may-be-a-victim-of-developaralysis/)，但它实际上是几个不同层面上的真实问题。为工作选择错误的工具意味着大量的技术债务。如果工具不适合这个问题。或者，如果它适合这个问题，但是您的组织中没有其他人知道这个工具，并且它有一个类似 K2 的学习曲线。或者花在决定使用哪种工具上的时间和精力比你使用它所获得的时间和精力要多。

所有这些都是在[不断增加的连通性](https://web.archive.org/web/20230129062556/https://www.quantamagazine.org/20150714-explosive-percolation-networks/) …和复杂性的大背景下发生的。Zeynep Tufekci 几个月前写了一篇关于复杂性和技术债务的[优秀文章](https://web.archive.org/web/20230129062556/https://medium.com/message/why-the-great-glitch-of-july-8th-should-scare-you-b791002fff03):

> 许多软件现在已经老到可以是多层的了…许多新代码编写得非常非常快…本质上，代码中有许多相当于“管道胶带”的东西，将事物结合在一起…随着软件吞噬世界，它进入了越来越复杂的情况，其中代码与其他代码、数据以及人在野外进行交互…这有点像知道自己患有慢性疾病，但假装你将面临的成本仅限于本月将面临的成本。

每个人都喜欢机器学习，它很热，很性感，一家又一家的初创公司声称这是他们的“秘方”和“护城河”——这是一个令人不快的形象，我们都同意——但与此同时，机器学习有时被描述为“[技术债务的高息信用卡](https://web.archive.org/web/20230129062556/http://research.google.com/pubs/pub43146.html)……由于其不透明和复杂，“在应用机器学习时，很容易在系统层面招致大量的持续维护成本”。

我发现自己又回到了理查德·马尔(Richard Marr)关于技术债务的这篇[聪明而务实的文章:](https://web.archive.org/web/20230129062556/https://medium.com/@richmarr/technical-debt-and-why-most-people-who-talk-about-it-are-just-plain-wrong-5c52b5d7121c)

> 科技债务既有成本也有价值…就像金融债务和自第一个夏普洛克以来的所有其他工具一样，这是一个你应该小心使用的工具…债务的价值来自早期交付。当存在产品不确定性时，它的价值最高…不同阶段的公司对债务有不同的容忍度…数据模型债务成本更高…语言和框架也可能是债务。

尤其是最后一句话，听起来越来越真实。我告诉你，工具和技术的寒武纪大爆发，加上我们日益复杂的系统互联性(在许多方面，它们本身都是很好的东西！)能够也经常让我们的集体技术债务增长到令人担忧的水平，就像美国的学生贷款一样。

当然，这并不意味着我们不应该使用新的工具、技术和框架。相反:我们应该渴望这样做。但是我们应该谨慎使用它们，因为它们本身的原因，和/或为了我们已经知道如何解决的问题。在 HFC，我们构建了许多 Rails 网站和 API，我们将一些现有的 Ruby gems 组合成一个[种子 gem](https://web.archive.org/web/20230129062556/http://seed.happyfuncorp.com/) ，用于大多数新项目，以快速处理“标准”web 管道，这样我们就可以找到有趣的问题。起初我对此有点怀疑——这似乎有点太一刀切了——但它的效果非常好。

我认为，教训是这样的:一般来说，作为一个经验法则，在第一个近似值上，你应该尽量使用新的工具和技术来解决新类型的问题，或者那些还没有很好解决的问题，而不是不断地尝试用另一种语言/框架来重做一切，希望这一次它会成为唯一的真正解决方案。正如软件中经常出现的情况，迭代，而不是革命，似乎是最明智的途径。
