发布四年后，Kubernetes 取得了长足的进步

原文：https://web.archive.org/web/https://techcrunch.com/2018/06/06/four-years-after-release-of-kubernetes-1-0-it-has-come-long-way/

2014 年 6 月 6 日，Kubernetes 首次发布。当时，没有人能预测到 4 年后这个项目会成为容器编排的事实上的标准，或者世界上最大的技术公司会支持它。那是以后的事。

如果你回想一下 2014 年 6 月，多亏了 Docker，容器化才刚刚开始起步，Docker 向开发人员普及了这一概念，但由于太早，没有标准的方法来管理这些容器。

谷歌多年来一直使用容器作为交付应用的方式，并运行一个名为 Borg 的工具来处理编排。它之所以被称为指挥器，是因为它就像管弦乐队的指挥一样，决定容器何时启动，以及一旦完成工作后何时关闭。

当时，两位谷歌工程师 Craig McLuckie 和 Joe Beda(他们后来创办了 Heptio )正在考虑为那些可能不具备谷歌工程人才深度的公司开发一款类似 Borg 的编排工具。他们想把这种如何开发分布式应用程序的想法传播给其他开发者。

你好世界

在第一个版本上市之前，后来成为 Kubernetes 的东西是出于对编排层的需要而开发的，Beda 和 McLuckie 已经考虑了很长时间。他们都参与了将 Google Compute Engine(Google 的基础设施即服务产品)推向市场，但他们觉得工具中缺少一些东西来填补基础设施和平台服务产品之间的差距。

“我们长期以来一直想找到一种方法，在生产中引入一种更进步、更协调的应用程序运行方式。McLuckie 解释说:“根据我们自己使用 Google Compute Engine 的经验，我们亲眼看到了企业在将工作负载迁移到云时面临的一些挑战。

他说，他们也了解与基于虚拟机的工作负载相关的一些限制，他们正在考虑使用工具来帮助解决所有这些问题。“所以我们想出了开始一个新项目的主意，这个项目最终成为了 Kubernetes。”

让我们开源吧

当谷歌在 2014 年 3 月开始开发 Kubernetes 时，它只想将容器编排带给大众。这是一个巨大的目标，McLuckie，Beda 和队友 Brendan Burns 认为实现这个目标的唯一方法是开源技术并围绕它建立一个社区。事实证明，他们的评估是正确的，但当时还不能 100%确定。没人能做到。

图片:云计算基础

“如果你看看历史，我们决定开源 Kubernetes，并使其成为一个面向社区的项目，这比传统智慧更快，并专注于以开放和参与的方式真正建立一个社区。这确实带来了回报，因为 Kubernetes 已经加速并有效地成为容器编排的标准，”McLuckie 说。

他们做的下一件事是创建云本地计算基金会(CNCF) 作为该项目的伞式组织。如果你仔细想想，这个项目可以有几个方向，正如现任 CNCF 导演丹·科恩在最近的一次采访中所描述的那样。

转向云原生

Kohn 说 Kubernetes 在几个方面是独一无二的。首先，它是基于谷歌多年来开发的现有技术。科恩说:“尽管 Kubernetes 的代码是新的，但其背后的概念、工程和技术诀窍是基于谷歌建造博格(以及失败的博格替代品 Omega 年的经验。”。另一件事是 Kubernetes 从一开始就被设计成开源的。

图片:Flickr 上的 Swapnil Bhartiya。根据 SA 2.0 许可证在 CC 下使用

他指出，谷歌本可以和 Kubernetes 在几个方向上合作。它本可以创造一个商业产品，并通过谷歌云销售。它可以开源它，但是有一个强大的中心领导，就像他们在 Go 上做的那样。他们可以去找 Linux 基金会，说他们想创建一个独立的 Kubernetes 基金会。但是他们没有做这些事情。

McLuckie 说，他们决定做一些完全不同的事情，并把它放在 Linux 基金会的赞助下，但不是作为 Kubernetes 项目。相反，他们希望为云原生计算本身创建一个新的框架，于是 CNCF 诞生了。“CNCF 是一个非常重要的舞台，不仅仅是对 Kubernetes 来说，而是对需要整合在一起以真正完成故事的技术来说，使 Kubernetes 成为一个更加全面的框架，”McLuckie 解释道。

让每个人都朝着同一个方向前进

在过去的几年里，我们看到 Kubernetes 已经成长为一个容器编排标准。去年夏天，一系列大型企业相继加入 CNCF，成为 T2 的 AWS，甲骨文的 T5，微软的 T7，VMware 的 T8 和 Pivotal 的 T9。他们与红帽、英特尔、IBM 思科和其他已经是成员的公司走到了一起。

云计算原生计算基金会白金会员

这些玩家无疑都想控制编排层，但他们看到 Kubernetes 的势头如此之快，他们别无选择，只能跟着走。科恩开玩笑说，让所有这些大牌球员加入就像养猫一样，但把他们带进来一直是我们的目标。他说事情发生得比他想象的要快得多。

在最近一次 TechCrunch 的采访中，在谷歌负责开源 Kubeflow Kubernetes 机器学习项目的 David Aronchick 在早期运营 Kubernetes。他对这种增长速度感到震惊。“我没想到会是这样。我于 2015 年 1 月加入，负责谷歌 Kubernetes 的项目管理。我对这种东西被压抑的需求感到震惊，”他告诉 TechCrunch。

随着它的成长，很明显 McLuckie 关于构建云原生框架而不是独立的 Kubernetes 基础是正确的。今天有几十个相邻的项目，这个组织正在蓬勃发展。

没有人比 McLuckie 自己更震惊了，他说看到 Kubernetes 自最初发布以来达到这些里程碑对他和他的团队来说是惊人的。“这只是一系列精彩的时刻，因为 Kubernetes 已经获得了足够的支持，看到社区真正团结在它周围真是太有趣了。”
