# 可怕的技术面试 

> 原文：<https://web.archive.org/web/https://techcrunch.com/2015/03/21/the-terrible-technical-interview/>

传统的技术面试对每个人来说都很可怕。对于公司来说，这是一种评估候选人的糟糕方式。对于候选人来说，这是一种评估公司的糟糕方式。它们浪费时间，给双方都带来压力。如果被追问，几乎每个人都会承认这一点。然而它们依然存在。

我谦卑地建议，现在是拥有选择余地的工程师们开始断然拒绝参与其中的时候了。

不要惊慌。我有更好的选择。请继续阅读。

上个月，丹尼·克莱顿写了几篇关于技术面试的[和](https://web.archive.org/web/20221222212500/https://techcrunch.com/2015/03/08/on-secretly-terrible-engineers/)[的](https://web.archive.org/web/20221222212500/https://techcrunch.com/2015/03/15/gray-beards/)优秀的帖子:你应该读一读，但让我来列举一些亮点:

> 很少有职业像软件工程这样公开地敌视他们当前的成员…我们希望人们在紧张的面试条件下在白板上做现场工程，因为，嗯，因为这是我们一直做的…在工程师紧缩的时代，我们根本不能放弃这么多人才。

反过来，他也受到了 T4 的启发:

> 软件开发员的工作面试不行。公司应该停止依赖他们。最精明的团队将通过设计替代性的招聘方案，在竞争中胜出。

确实如此。有趣的是，我确实有这样的印象，事情终于发生了变化。越来越多的公司要求应聘者做测试项目，而不是白板面试。其他人对在面试阶段消除假阳性越来越不热衷(但对几个月后解雇他们越来越无情。几年前，谷歌的人力资源主管[承认](https://web.archive.org/web/20221222212500/https://techcrunch.com/2013/06/22/the-technical-interview-is-dead/)“脑筋急转弯完全是浪费时间”和“考试分数毫无价值”，这一点很有帮助。

然而。几年前，我可能雄心勃勃地写过“[技术面试已死](https://web.archive.org/web/20221222212500/https://techcrunch.com/2013/06/22/the-technical-interview-is-dead/)”，但事实并非如此。至少现在还没有。它正在死去，但是太慢了。

我们这些讨厌的人需要承认一点:公司坚持白板式面试是有原因的，其中一些是半有效的，即使他们知道他们远非完美。其中包括:

*   从公司的角度来看，面试是一个相对可测量和可重复的过程。每个人都知道他们是如何工作的。只要提出几个问题，和几个衡量答案的标准，在紧要关头，(几乎)任何技术员工都可以进行面试。
*   公司希望尽早筛选掉明显不合适的候选人，很难抑制这种感觉，即当你在面试时，你还不如问他们一两个稍微技术性的问题……这些问题会在一场全面的传统面试中很快出现。
*   有天赋这种东西，你确实想过滤掉没有太多天赋的人。传统的技术面试被认为更容易出现假阴性而不是假阳性。从历史上看，误报被视为灾难场景；雇佣一个糟糕的工程师被认为比雇佣两个好工程师更糟糕。但如今优秀的工程师如此稀缺，这种情况已经不再适用。
*   指派一个测试项目——目前技术面试的替代方案——充其量也是不完美的。实际上，想出一个既有意义又只占用候选人一天或几天时间的真正的小项目是相当困难的。与此同时，候选人希望在这段时间获得报酬，并且/或者抗议他们已经有工作，不能在一个投机项目上花费那么多精力，而公司则担心项目可能会被抄袭甚至外包。
*   这就是为什么个人推荐信和推荐信仍然是每个人最喜欢的招聘技巧…
*   …这反过来也是为什么科技行业的多样性数字如此糟糕的主要原因。

所以:如果我们*能够找到一种可靠的方式来替代传统的技术面试，这对公司、对候选人都有好处，而且有助于增加那些目前被推荐过程隐性忽视的代表性不足的群体的数量。这对每个人来说都是一个巨大的双赢。*

如果我没有提到有很多创业公司正在尝试做这些事情，那我就是失职了。但是我有不同的想法。这让候选人承担了更多的责任……但我认为这是好的一面。我提议:

*   是时候让工程师们——尤其是需求很高的优秀工程师——开始断然拒绝接受白板面试了。
*   是的，真的。没有什么比在面试前失去有吸引力的候选人更能迫使公司更快地转向更好的技术了。
*   *但是。*这种拒绝必须伴随着反提议:用讨论/演练候选人在自己的时间里完成的一个附带项目来代替面试。

以下是我脑海中的流程:

1.  面试官需要 30-60 分钟来熟悉应聘者的项目
2.  然后，他们花一两个小时讨论项目、候选人做出的架构和实现决策、他们本可以选择的备选方案、他们想要添加的功能、代码的结构和逐行质量、环境和配置问题等。
3.  这些一般性的讨论主题被形式化了，这样它们就可以在面试中重复，候选人和面试官可以被比较，结果也可以被衡量。他们使用全局变量吗？代码是否遵循一个 [MVC](https://web.archive.org/web/20221222212500/http://en.wikipedia.org/wiki/Model%E2%80%93view%E2%80%93controller) 或者其他架构模式？方法的结构和封装是否合理？用户界面是否表明对可用性问题的理解？有测试代码吗？等等。半个通用代码评审；一半“候选人似乎真正理解了他们(声称)编写的这段代码多少”？
4.  然后面试官让应聘者实时地给他们的项目添加一个小的新功能。
5.  在这一点上，面试官应该完全相信(或完全怀疑)这个项目是否构建良好，以及候选人是否真的自己构建了它。
6.  如果是前者，那么在约定的时间，让候选人分支公司预先定义的测试项目——可能是一个单一的长期项目，也可能是每隔几个月就有一个新的项目。(对于新员工来说，建立一个新的是一个很好的项目。)然后让他们提交一个新功能的“拉”请求，这需要大约 4-8 个小时的工作。一个测试项目，各种各样，但相当小，只是作为一个健全的检查，并确保候选人可以以合理的速度工作。
7.  让不同面试者评估拉动式请求，这样你就可以从多个角度了解候选人。
8.  或者，另一种可能更有效的方法是，让候选人和不同的面试官搭档一两个小时，编写一个小程序。
9.  然后，决定是否录用。

瞧，这是技术面试的替代品，没有白板编码，没有陷阱问题，没有对算法实现细节的深入了解，候选人再也不用写了。我并不认为这是所有人的完美最终解决方案——但我确实相信，对于大多数仍在进行白板面试的公司来说，它或类似的东西将是一个可行的、比现状好得多的替代方案。

现在，这个*需要一个巨大的先决条件:每个候选人必须有一个他们自己写的副业，作为他们的名片。*

 *我不认为这是不合理的。事实上，我认为你可以非常愉快地过滤掉任何一个*没有*这样名片的人。(为了避免被指责说我只说不做:我很高兴被[聘为](https://web.archive.org/web/20221222212500/http://happyfuncorp.com/#home)全职软件工程师；我经常旅行，我[写书](https://web.archive.org/web/20221222212500/http://www.rezendi.com/)，还有这个 here 周刊 TechCrunch 专栏；我仍然有时间做我自己的软件项目。[下面是我最新的](https://web.archive.org/web/20221222212500/http://www.vortext.co/)、[开源](https://web.archive.org/web/20221222212500/https://github.com/rezendi/vortext)。)

四年前，当我第一次开始在这里抱怨传统软件面试的无效反生产性时，我写了[:*不要面试任何没有完成任何事情的人。永远不会。*证书和学位不是成就；我指的是有真实用户的真实项目。对于没有网站、应用程序或服务的软件开发者来说，他们没有借口说:“这是我自己做的！“在这个世界上，谷歌应用引擎(Google App Engine)和亚马逊网络服务(Amazon Web Services)都有免费的服务层，注册成为安卓开发者并在安卓市场上发布一款应用只需 25 美元。”](https://web.archive.org/web/20221222212500/https://techcrunch.com/2011/05/07/why-the-new-guy-cant-code/)

我认为今天更是如此。但另一方面，如果你确实有一项成就，一个可以指向的宠物项目，那么你就不应该跳过毫无意义的白板编码面试。你应该做得更好。我们都是。*
