Docker 屈服于不可避免，并提供本地 Kubernetes 支持

当谈到容器编排时，似乎很明显，由 Google 开发的开源工具 Kubernetes 已经赢得了运营的人心。因此，对于任何关注 Docker 的人来说，今天在哥本哈根的 DockerCon Europe 上宣布对 Kubernetes 的本地支持都不会感到意外。

该公司尚未完全放弃自己的编排工具 Docker Swarm，但通过首次提供原生 Kubernetes 支持，它承认人们正在大量使用它，他们必须建立支持。为了减轻支持竞争对手工具的痛苦，他们提供了一种架构，使用户能够在运行时选择编排引擎。Docker 的产品负责人 Banjot Chanana 告诉 TechCrunch，每次都可以使用 Swarm 或 Kubernetes，而无需修改代码。

在今天的声明之前，虽然可以在 Docker 上使用 Kubernetes，但这不一定是一个简单的过程。有了新的 Kubernetes 支持，对于 Docker 企业版和 Docker 开发者版用户来说，这应该简单得多。

Chanana 说，由于 Docker 的架构方式，在 Docker Swarm 的基础上提供 Kubernetes 实际上并没有那么困难，而且它不会看起来或感觉起来像一个插件。Docker 为客户提供了一种构建程序容器的标准方法。这通常由 DevOps 模型中的开发人员负责。

运营部门使用编排工具在容器的整个生命周期中对其进行部署、保护和管理。在过去的几年里， Kubernetes 作为编排工具的选择越来越受欢迎，像 AWS 、甲骨文、微软、 VMware 和 Pivotal 等大牌都在今年加入了云原生计算基金会，这是一个包含 Kubernetes 项目的开源组织。

当所有这些组织都追随潮流时，Docker 别无选择，只能顺应客户的愿望。Docker 能够在保持对自己的编排工具的支持的同时构建支持，但很明显，Kubernetes 已经成为人们未来将用于大多数容器工作负载的编排工具。

值得注意的是，本周的信息报道称，在 2014 年开发 Kubernetes 时，谷歌提出与 Docker 合作，并让其托管 Kubernetes 项目，但该公司决定开发 Swarm，谷歌转向了云原生计算基础。从某种意义上来说，今天的公告让他们兜了一圈，因为他们将支持 Kubernetes 向前发展(即使他们不存放代码)。
