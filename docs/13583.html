微软如何将 SQL Server 引入 Linux TechCrunch

原文：https://web.archive.org/web/https://techcrunch.com/2017/07/17/how-microsoft-brought-sql-server-to-linux/?ncid=rss

回到 2016 年，当微软宣布SQL Server 将很快在 Linux 上运行时，这个消息对用户和专家来说是一个重大的惊喜。在过去的一年中，微软对 Linux(以及一般的开源软件)的支持已经变得更加清晰，该公司现在的使命似乎就是把它的工具带到用户所在的地方。

该公司今天发布了第一个候选版本 SQL Server 2017 ，这将是第一个运行在 Windows、Linux 和 Docker 容器中的版本。仅 Docker 容器就已经有超过 100 万次的点击，所以毫无疑问，人们对这个新版本很感兴趣。虽然这个新版本中有许多新功能和速度改进，但 SQL Server 2017 支持 Linux 的事实仍然是这个版本最有趣的方面之一。

在今天的发布会之前，我与微软数据库系统集团的总经理 Rohan Kumar 进行了交谈，了解了更多关于这个项目历史的信息，以及他的团队如何设法将 SQL Server 这样一个极其复杂的软件带到 Linux 上。Kumar 已经在微软工作了超过 18 年，他指出他的团队注意到许多企业开始使用 SQL Server 来处理他们的关键任务工作负载。但与此同时，他们也在包含 Windows Server 和 Linux 的混合环境中工作。对于许多这样的企业来说，不能在 Linux 上运行他们选择的数据库成为了一个摩擦点。

“与企业交谈，很明显这样做是必要的，”库马尔说。“我们强迫客户使用 Windows 作为他们选择的平台。”在微软的另一个化身，这可能会被视为积极的事情，但该公司今天的战略是完全不同的。

Kumar 还指出，许多企业正在寻找甲骨文数据库产品的替代品。如果您想运行 Linux 并使用具有完全企业支持的专有关系数据库，毕竟您没有被宠坏的选择。

正如 Kumar 告诉我的，这不是他的团队第一次考虑 Linux 支持。“我们过去进行过几次讨论，但未获批准，”他告诉我。“这不被认为是企业的一种战略方式。”但是三年前——现在塞特亚·纳德拉已经是公司的最高领导人——这个团队决定再次提出这个想法。“最令人惊讶的是，我们预计会有很多来回。库马尔说:“看到决策制定得如此之快，真的令人惊讶。

在雷德蒙的一家越南餐馆里，坐在一碗河粉旁，这个团队做出了继续这个项目的最终决定。但是做出这个决定后，团队现在面临一个艰巨的任务:如何将数千万行 SQL Server 代码移植到 Linux 上？Kumar 也不想在功能上做出任何妥协，所以它要么必须是 SQL Server 的完整核心，要么什么都不是(目前，这不包括该公司在 Windows 上提供的图形用户界面和工具)。

团队在微软已经存在的一个项目中找到了答案:吊桥。Drawbridge 是早在 2011 年启动的一个研究项目，它基本上提供了一个具有小型 API 表面的容器和一个基本版本的 Windows，配置为在容器中高效运行应用程序。这里的想法基本上是构建更好、更安全的虚拟机。然后，库操作系统执行应用程序，处理内存管理和其他重要功能，并与底层操作系统集成。

大约两年前，SQL Server 团队决定将此作为其 Linux 工作的核心。“领导层表达了适当的关注，”库马尔评论道——我猜测，鉴于吊桥在很大程度上是一个实验项目，人们确实有相当多的担忧。但是 SQL Server 团队接管了 Drawbridge 代码库，并将其添加到 SQL OS 层。

从很多方面来说，这个操作系统层使得这个项目成为可能。因为 SQL Server 的需求总是超出 Windows 和 Windows Server 所能提供的范围，特别是在内存管理方面，所以团队已经将许多标准操作系统功能内置到 SQL Server 的操作系统层中。例如，得益于此，Drawbridge 中的 SQL Server 也可以管理自己的内存。这方面的工作非常成功，该团队不仅为 Linux 上的 SQL Server 构建了这个，而且实际上将 SQL OS 和它在 Drawbridge 上所做的工作合并到新的 SQL 平台抽象层中，该抽象层现在可以在 Windows 和 Linux 上运行。

因此，SQL Server 团队可以从单个代码库开始工作，而不必担心代码将在哪里运行(这包括微软的 Azure 平台)。

SQL Server for Linux 应该会在今年晚些时候全面上市。即使在今天，一些公司已经在生产中使用它，Linux 版本现在运行速度和 Windows 版本一样快(假设硬件相当)。

展望最终版本，Kumar 指出，团队将密切关注下一步该做什么。尽管数据库领域的创新不断加速，但并不是所有的微软客户都希望他们的关键任务系统每年更新一次(甚至更快)。鉴于我们最近已经看到了 SQL Server 2016 和 2017 的年度 SQL Server 版本，如果我们没有尽早听到第一个 SQL Server 2018 预览版，我会感到惊讶。
