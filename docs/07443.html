初创公司最常犯的营销错误

我在世界上一些最好的软件公司看到过同样的基本营销错误:如果你的营销团队需要工程人员的帮助来更新他们的网站(发布新帖子、编辑文案、上传图片等)。)，那么他们已经被设定失败了。

当公司高管(通常是工程师本身)不熟悉他们雇佣的营销人员、网站设计师和开发人员的日常工作时，最容易犯这种错误。我知道有几亿甚至几十亿美元收入的软件公司，营销人员根本无法完成他们的工作，不得不等待几天/几周/几个月的工程来更新他们的网站。

发起新的在线活动可能意味着等待公司的工程团队取消对 产品功能 的工作优先级，以便为营销需求创造时间。它可以成为大规模的组织功能障碍。

这太疯狂了(一次又一次地犯同样的可预见的错误),应该停止了。现在是2016年——我们不应该还在进行这场辩论。

在这篇文章中，我想深入探讨这个问题是如何产生的，为什么它会被忽视这么久，以及营销人员(和开明的初创企业高管)可以做些什么来解决它。

什么是网站？

你的公司做的第一笔营销投资是建立你的网站，这是有原因的。当公司开始运作时，他们通常会在印刷名片之前建立一个网站。你的网站实际上是你公司的门面——世界上的任何人都可以通过互联网即时访问它，从而将他们置于任何数字营销工作的战略中心。出于这个原因，花在网站上的钱(1900亿美元)比所有数字广告(1540亿美元)都多。

早期创业公司的网站通常被视为(和建造)产品的延伸——采用平面HTML/CSS，部署在与公司核心产品相同的代码路径上。当设计和制造产品的同一个团队自己承担所有营销责任时，这是有意义的。

但总有一天，你需要成长起来，雇佣营销人员，建立一个合适的网站。我知道很多收入达5000万美元的创业公司，网站更新需要重新部署他们的核心产品。这意味着每次营销部门想要更新他们的网站副本时，他们必须说服工程部门进行新的部署。

一个优秀的营销人员会专注于信息传递、写作、品牌、搜索引擎优化、结构方程模型、分析等。他们中有些人会编码，有些人是分析工程师——但他们不是软件工程师(他们是营销人员！).要求营销人员学习如何提交GitHub Pull请求，并通过你的工程审批渠道来完成他们的工作，这是一种过分的做法。你现在不必要地限制了他们的工作，给他们的工作带来了不必要的挑战。

不能控制内容的营销人员无法完成他们的工作。

如果你拖延太久，并且将这种情况发挥到极致，你最终会得到一个由几十名员工组成的营销团队，他们自己无法更新和控制内容。在一个工程团队中，为了部署代码，你不得不在拷贝编辑队列中排队等待，以获得营销部门对代码注释的语法和句子结构的批准，你会喜欢这样的工作吗？

让工程师编码，让营销人员发布内容！不能控制内容的营销人员无法完成他们的工作。

进入内容管理

简而言之，内容管理软件使营销人员能够通过图形界面自行更新网站内容。没有编码，没有拉取请求，没有代码审查——不需要工程干预。内容管理起作用了。内容管理使营销人员能够完成他们的工作。

内容管理软件已经存在了20多年，但奇怪的是，对于一个如此大(1900亿美元)的行业来说，它只是在最近才成熟起来。 Drupal 和 WordPress 现在占据了65%的市场份额(很快将达到80%)。WordPress和Drupal赢得了市场，主要是因为专业网站设计者和开发人员的行业基本上已经标准化了他们的软件工作。在过去的几年里，如果你是专业的网站开发人员，除非你使用Drupal或WordPress，否则你不会被营销人员雇佣。

到目前为止，大多数营销人员都知道如何使用这些工具，并且使用起来很舒服(问问他们就知道了！).有一个越来越强大的供应商生态系统(泛神论与之竞争)，专门从事WordPress和Drupal网站的运营——托管、扩展、调整和开发。但它是一个有自己的工具和供应商的专业行业，与更广泛的软件工程行业有联系但又分离。我发现许多初创公司的工程师被吸引到网站项目中来，但通常不了解内容管理行业，这也是造成这种脱节的原因。

初创公司的网站技术决策是如何下降的

这听起来耳熟吗？

  • 预埋 : 本公司产品的工程及产品设计团队拥有本网站。它通常是挂在产品代码基础上的XHTML/CSS。生活很简单，工程师自己管理内容。这个管用。
  • 种子阶段 : 你的员工中有了你的第一个营销人员。你试着教他们如何更新网站。这很奇怪，但还算管用。你知道你正在阻碍他们，但是谁有时间去实现一个全新的网站呢？
  • A系列 : 您现在正在扩大您的营销团队。你有一个营销主管，他不停地提到网站，以及他们自己怎么也更新不了。他们想要WordPress。你突然想起上一个WordPress网站需要管理多少PITA，于是你恳求道:“我明白，但是我不认为我们现在可以优先考虑这个。”
  • 缩放 : 你现在已经建立了一个成熟的营销团队。内容营销者、作家、编辑、设计师、需求生成，所有的一切。营销团队已经放弃拥有该网站。你会不时听到一些奇怪的抱怨，但是“事情就是这样。”你的产品团队(设计师和工程师)必须深入参与任何新的营销活动。事情似乎进展得比预期的要慢得多，但重建网站的想法让所有参与其中的人都感到力不从心。在这一点上，没有人想拥有它。
  • 然后 : 某天晚上你正在浏览竞争对手或姐妹公司的网站，你意识到你有多讨厌自己的网站。你的网站可能设计得很好，但内容却少得令人难以置信。你知道它不能公平地反映你的产品和公司的质量和深度。你知道demand-gen正在遭受损失，你也能明白为什么——在一个合适的网站上发布的数量和质量令人难以置信。你正坐在一辆(令人惊叹的)1967年的卡玛洛里和一辆特斯拉赛车进行飙车比赛。你的网站乍一看可能很酷，但是你已经被远远地甩在了后面。

即使你确信营销需要内容管理，也不是你打个响指就能搞定的。让我们来看看在管理你公司网站时发生的真正的讨价还价。让我们确保营销得到适当的武装，因为让一个营销人员与你的世界级工程师在技术决策上进行权衡并不总是公平的。

工程师试图解释为什么不需要内容管理软件的主要原因(以及告诉他们什么)

我们不能运行WordPress或Drupal，因为它们不安全。

这是一个很懒的借口。虽然你确实需要管理Drupal和WordPress的安全更新，但工程师负责的所有软件都是如此。这就像一个营销人员说“我们不能投资公关，因为这可能会导致糟糕的公关。”技术上是对的，但仍然是错的。

工程师可能真的不想自己承担安全责任，这是公平的，但是有许多很好的选择来外包这项责任。

对于你的安全工程，我的后续问题是:我们真的想在与我们的产品和用户数据相同的基础设施上运行我们的公共、可访问互联网的网站吗？将您的网站从您的产品中分离出来可以通过深度安全来提高安全性。

我们的网站开发者不想使用WordPress或Drupal。

如果你从你的产品团队中借用工程师来开发你的网站，这种情况经常发生。以构建产品为生的工程师更熟悉软件工程师的工具——React、Ruby on Rails——而不是专业网站开发人员使用的工具，如WordPress和Drupal。如果这些开发人员在产品和营销部门之间共享，这可能会特别棘手；很难要求工程师在两种截然不同的环境之间来回切换。

总有一天，你需要长大，雇佣营销人员，建立一个合适的网站。

然而，公平地说，对于营销来说，你是在进行一种隐性的交易。你需要坐下来决定什么更重要:是让你的前端工程师使用他们喜欢的工具，还是让你的营销团队能够管理他们的内容。这是一个需要深思熟虑权衡利弊的决定。

长期来看，很明显，这里真正的答案是获得营销他们自己的资源来开发网站。越快越好，因为产品团队越来越难抽出时间来帮助营销；当然，他们有自己“真正”的工作要做。

我用来更新个人网站的平面文件网站技术比Drupal和WordPress好得多，所以我们应该使用它。我来教你怎么做拉请求，很简单！

这纯粹是狂妄。评估营销软件的专家是营销人员，而不是你的工程团队。如果你的营销人员走到你的首席技术官面前，告诉他们“我去参加了一个会议，看到了一个来自Oracle的数据中心管理软件的演示，这个软件非常强大，会发生什么呢？我真的担心我们会错失良机。”

如果一名工程师告诉你的营销主管他们知道如何更好地评估营销技术，你的营销人员应该礼貌地微笑并保持坚定(并随时将他们发送到本文)。

WordPress和Drupal有些过头了，我们的工程师将会为你创建一个定制的CMS，它会工作得更好。

这是超越狂妄的。真的很蠢。任何计划花费数十万美元(更不用说数百万美元)来构建一个定制的CMS的人都是徒劳的。我没有低估风险。在极端情况下(例如在数字出版商)，我看到过这个决定让公司脱轨。

如果您的CTO得出结论，为了构建您的产品，他们需要开发自己专有的Linux内核替代品，您会怎么说？

WordPress和Drupal是巨大的、成熟的开源项目，有数千名贡献者和由数十万专业开发者、营销用户和供应商组成的生态系统。就成功的开源项目而言，它们与Linux内核不相上下。他们利用了数亿美元的开源工程成果。它们是非常坚固的产品。

你的工程团队可能很有才华，但他们不会在一夜之间开发出WordPress或Drupal的替代品。他们将为你建立一个复杂的、漏洞百出的网站钱坑，你的营销部门将无限期地与之结婚。对于营销人员来说，这可能是职业终结。

如果你的工程团队得出了这个结论，你需要督促他们——你的公司不能犯这样的错误。

我们现在对当前的网站投入太多，没有办法立即优先重建它。

这是一个艰难的决定，因为对贵公司来说，这是一个高层次、高风险的优先决策。

你首先需要回答对于你的公司，我们的网站有多重要？我会从找出你网站上的销售线索的百分比开始这个练习。(对于软件公司来说，这一比例通常高达95%。)

然后你需要权衡重建网站的成本(时间和金钱)和拥有一个高效网站的机会成本。公平地说，您将需要权衡产品化本身的项目风险，因为这些项目可能很复杂。

最后，你需要回答:如果我们现在不重建我们的网站，那什么时候呢？
缓慢但肯定理智的商业决策会赢得胜利。
